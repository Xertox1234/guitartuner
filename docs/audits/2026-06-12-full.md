## High Findings

### H1 — Bass Lock Never Fires: 0.9 Confidence Gate Unreachable for Low Strings
**File:** `App/Engine/LiveTunerModel.swift` (line 66); `TunerEngine/PitchReading.swift` (line 51)
**Introduced by:** 6c79875 (play confirmation ping)

`minLockConfidence = 0.9` is compared against `PitchReading.confidence`, which is `det.clarity` (the raw NSDF peak value). The pipeline's own comment documents inharmonic-low bass clarity as "~0.7–0.85 on raw frames". This means low strings — bottom two on 6-string, entire instrument on 5-string — will routinely fail the confidence gate, silently preventing both the confirmation ping and the lock haptic from ever firing on those strings.

**Before fixing:** Check the clarity distribution in `BenchmarkSuite` for E2 and B0. The `SustainGate` may filter before emitting, pushing the published value higher than raw-frame clarity. If not, the right fix is a per-band threshold in `AnalysisConfig` (e.g. `0.75` for `f0 < 120 Hz`), not a flat lowering of `0.9`.

### H2 — Confirmation Ping Fires Rapidly Near Lock Boundary — No Hysteresis
**File:** `App/Engine/LiveTunerModel.swift` (lines 223–228)
**Introduced by:** 6c79875 (play confirmation ping)

`handleLock` triggers on every false→true `wasLocked` transition with no cooldown or hysteresis. `wasLocked` is cleared as soon as `locked == false`, so any single-frame dip out of lock (from cents oscillation near ±3¢, or a confidence dither during sustain decay) rearms the trigger. The result is rapid-fire pings and haptics when hovering near the threshold — audible stutter on string settling.

**Fix:** Add a minimum inter-ping cooldown (1.5–2 s) tracked as `@ObservationIgnored private var lastLockDate: Date` in `LiveTunerModel`. Gate the ping (not the haptic) on `Date().timeIntervalSince(lastLockDate) > 1.5`.

### H3 — `ToneGenerator.start()` Reconfigures Live `AVAudioSession` Mid-Capture
**File:** `App/Engine/ToneGenerator.swift` (lines 52–57, 69–99); `App/Engine/LiveTunerModel.swift` (line 226)
**Introduced by:** 6c79875 (play confirmation ping)

`tone.ping()` lazily calls `ToneGenerator.start()` on the main actor at lock time. `start()` synchronously calls `AVAudioSession.setCategory(.playAndRecord, mode: .measurement)` + `setActive(true)` while `AudioCapture` has already configured the session as `.record`. On iOS this is a live category change on a running capture session — it can silently deactivate and reactivate the session, removing the installed tap and stopping pitch detection. `AudioCapture` has no interruption handler to restart.

**Fix:** Pre-start `ToneGenerator` eagerly in `LiveTunerModel.start()` alongside `haptics.prepare()`, before `TunerEngine.start()`. The `AVAudioSession.setCategory(.playAndRecord)` call must be made once before capture starts — either in `LiveTunerModel` (as the session owner) or in `ToneGenerator.prepare()`. `AudioCapture` must not race it.

### H4 — Confirmation-Ping Logic Has No Test Coverage
**File:** `App/Engine/LiveTunerModel.swift` (lines 221–229)
**Introduced by:** 6c79875 (play confirmation ping)

`handleLock` contains three behaviors with no tests: (1) ping fires only on rising edge, (2) ping suppressed when `toneOn`, (3) ping re-arms after watchdog timeout. `LiveTunerModel` stores concrete `TunerEngine`, `ToneGenerator`, and `LockHaptics` instances with no injection point, making it untestable headlessly. No `LUMATests` Xcode target exists in `project.yml`.

**Fix:** Extract a `LockEventHandler` protocol wrapping the ping/haptic calls and inject it in `LiveTunerModel.init`. A fake implementation enables headless tests covering all three conditions.

### M2 — Lock Bloom Color Diverges from All Other Lock Affordances Under Non-Default Palettes
**File:** `LumaDesignSystem/Strobe/AuroraStrobe.swift` (line 139); `RadialStrobe.swift` (line 151); `MetalStrobe.swift` (`colTune` uniform)
**Introduced by:** 89ed14c (wire LumaPalette through strobe renderers)

Under non-aurora palettes, the strobe bloom adopts the palette's tune hue (amber, neon, etc.) while every other lock affordance (`.lumaGlow`, `ReducedGauge`'s ring, `StringRow` lock indicator) remains fixed-mint. The visual incoherence is only visible for non-default palette users.

**Fix:** Product decision needed: either (a) accept the divergence and document it, or (b) derive `LumaColor.tune` / `LumaGlow` from the active palette environment so all lock affordances move together.

### M3 — `minLockConfidence` Hardcoded in Two Unlinked Places
**File:** `App/Engine/LiveTunerModel.swift` (line 66); `TunerEngine/PitchReading.swift` (line 51)
**Introduced by:** 6c79875 (play confirmation ping)

`LiveTunerModel.minLockConfidence = 0.9` and `PitchReading.isLocked(minConfidence: Double = 0.9)` are currently equal but not linked. A change to one that misses the other creates a silent behavioral asymmetry between auto and lock modes. (Also see H1 — the threshold itself may be too high for bass.)

**Fix:** Define a single `lockConfidence` constant in `LumaMusic` alongside `lockCents` and use it at both sites.

### L1 — Spring Animation Keys Only on `active`, Not `locked`
**File:** `LumaDesignSystem/Components/StringRow.swift` (lines 85–86)
**Introduced by:** f5f71ef (spring-pop animation)

Scale expression is `active || locked ? 1.0 : 0.95` but animation is `.animation(.spring(...), value: active)`. If `locked` transitions on a cell where `active == false`, scale snaps without animation. Latent in current app logic (`lockedIdx == activeIdx` always) but structurally incorrect.

**Fix:** Key the animation on `value: active || locked`.
