## Critical Findings

### C1 — CI benchmark gate contradicts committed accuracy spec
**File:** `Packages/TunerEngine/Sources/Benchmark/main.swift:81`
**Domain:** testing
**Status:** new

The CI gate at line 81 fails if `worstAbsCents > 2.5`. The committed accuracy.md records the current measured worst-case as **3.79¢** (harmonic signal, bass range). CI runs `Benchmark --compare --out docs/benchmarks --ci` (ci.yml line 32-33), so this gate is active on every push. The gate comment reads `// P2r: ~1.72¢`, indicating the threshold was written anticipating a P2r improvement that has not yet landed. Either CI is currently failing on every push, or the benchmark produces different numbers on the Linux runner due to the scalar fallback path. Either way the gate is miscalibrated: it must be relaxed to `≥ 4.0¢` (above 3.79¢ + margin) until P2r lands, or the spec must be updated.

### C2 — Missing triple-buffer semaphore in StrobeRenderer — GPU/CPU sync absent
**File:** `Packages/LumaDesignSystem/Sources/LumaDesignSystem/Strobe/MetalStrobe.swift:184`
**Domain:** strobe
**Status:** new

`StrobeRenderer.draw(in:)` submits a new `MTLCommandBuffer` every frame with no `DispatchSemaphore` to pace CPU ahead of GPU. The rule mandates a semaphore initialized to 3, with `signal()` in the `addCompletedHandler`. Without it the CPU can enqueue frames faster than the GPU consumes them, causing the driver to stall inside `commit()`, producing frame-pacing jitter visible at 120fps. There is also no `addCompletedHandler` at all in this renderer.

## High Findings

### H1 — Bloom activates without confidence gate — false-lock visual degrades lock moment
**Files:** `Packages/LumaDesignSystem/Sources/LumaDesignSystem/Strobe/AuroraStrobe.swift:73`, `RadialStrobe.swift:70`, `MetalStrobe.swift:207`
**Domain:** strobe
**Status:** new

All three renderers compute `inLock = input.locked || abs(err) < LumaMusic.lockCents`. The `|| abs(err) < LumaMusic.lockCents` fallback bypasses the confidence gate baked into `input.locked` in `LiveTunerModel` (which requires confidence ≥ 0.9 for mid/high, 0.75 for bass). During the attack of a bass note, confidence can be 0.70–0.85 while cents transiently pass through 0, triggering bloom with no haptic — a visible desync that degrades the product's emotional core. Fix: use `inLock = input.locked` exclusively in all three renderers and remove the fallback clause.

### H2 — StrobeInput contract divergence: wrong types, missing isIdle and TunerState
**File:** `Packages/LumaDesignSystem/Sources/LumaDesignSystem/Strobe/StrobeInput.swift:14`
**Domain:** strobe
**Status:** new

The canonical `StrobeInput` contract specifies `phase: Float, cents: Float, state: TunerState (.flat | .sharp | .inTune | .idle), isIdle: Bool`. The actual implementation has `phase: Double, cents: Double, locked: Bool` — no `isIdle`, no `TunerState`. The idle flag is passed as a separate constructor parameter on each renderer rather than being inside `StrobeInput`, making it easy for a new renderer to silently omit idle handling.

### H4 — Heap allocation on audio render callback thread (scratch buffer resize)
**File:** `Packages/TunerEngine/Sources/TunerEngine/Capture/AudioCapture.swift:78`
**Domain:** capture
**Status:** new

`if scratch.count < frames { scratch = [Float](repeating: 0, count: frames) }` performs a conditional heap allocation inside the audio render callback. Even though it fires only when frames grows, the audio render thread rule is unconditional: no allocation, ever. A resize can block on `malloc` and cause a priority inversion with the RT scheduler. Fix: pre-allocate `scratch` to the maximum expected frame count during `AudioCapture.start()` before the tap is installed.

**Domain:** capture
**Status:** new

`out.baseAddress!` is a force-unwrap on the real-time audio thread. A crash here is unrecoverable. Fix: `guard let base = out.baseAddress else { return }`.

### H6 — BenchmarkTests accuracy tolerances provide no meaningful CI protection
**File:** `Packages/TunerEngine/Tests/TunerEngineTests/BenchmarkTests.swift`
**Domain:** testing
**Status:** new

`testAllMethodsScoreAccurately` asserts `meanAbs < 12` — 48× looser than the 0.25¢ CI gate. `testCaseRunnerScoresCleanTone` asserts `meanAbs < 3.0` — still 12× above the gate. ### H7 — `testCaseRunnerScoresCleanTone` lockStats always empty — lock-time assertion never executes
**File:** `Packages/TunerEngine/Tests/TunerEngineTests/BenchmarkTests.swift`
**Domain:** testing
**Status:** new

## Medium Findings

### M1 — `spectralRefineMinHz` duplicates `AnalysisConfig.midLowHz` — dual source of truth
**File:** `Packages/TunerEngine/Sources/TunerEngine/Pipeline/PitchPipeline.swift:43`
**Domain:** dsp
**Status:** new

`PitchPipeline` declares `static let spectralRefineMinHz: Double = 120` independently of `AnalysisConfig.midLowHz = 120`. Both represent the same physical boundary. If either is changed independently, the refine path and the window-band selection will disagree. Rule: `AnalysisConfig` is the single source of truth.

### M2 — Multiple load-bearing gate thresholds hardcoded outside AnalysisConfig
**File:** `Packages/TunerEngine/Sources/TunerEngine/Pipeline/PitchPipeline.swift:37` and `DSP/Smoothing.swift`
**Domain:** dsp
**Status:** new

Five tuning constants governing voiced/unvoiced gating and lock detection are scattered outside `AnalysisConfig`: `lockPrecisionThreshold = 1.0` (line 37), `SustainGate(minConfidence: 0.6)` overriding the default 0.7 without documentation, `emitFloor = 0.5`, `alpha = 0.35` and `snapCents = 120` in `FrequencySmoother`. Additionally, `k = 0.9` (NSDF peak-selection) is hardcoded in `PitchDetector.swift:91`. All of these are accuracy-critical constants. Inlining them makes test overrides and cross-constant reasoning fragile.

### M3 — No post-NSDF octave-history guard — single point of octave safety
**File:** `Packages/TunerEngine/Sources/TunerEngine/DSP/PitchDetector.swift:91`
**Domain:** dsp
**Status:** new

`PitchPipeline` stores `trackedFrequency` but never checks the NSDF-detected frequency against it before proceeding to refinement. The only defense against an octave jump in pure `.mpm` mode is the `k = 0.9` threshold. If MPM picks an octave alias (peak value ≥ 0.9 × nmax), it passes through all refine stages with no secondary gate. The current 0.00% octave-error rate is achieved with k=0.9; the architecture has no recovery for pathological real-world signals not in the benchmark suite. A `|1200·log2(f/tracked)| > 1200 && clarity < 0.95 → handleUnvoiced()` check would provide defense-in-depth.
